WEEK 01

The Anatomy of a Voice Agent

MILESTONE M1 · TALKBOX — STT → GROK → TTS, THIS WEEK

Voice is the hardest UI

By week 6 you'll have built an AI you can talk to — and interrupt.

Open with a 30-second live demo of the FINISHED project, including a barge-in. Then: "everything you just heard is a lie held together by buffering. Let's take it apart."

What you'll build

  • M1 Talkbox — push-to-talk Grok agent: STT → Grok → TTS, this week
  • M2 Plumber — continuous streaming + a simulated phone line
  • M3 Listener — drop the button: VAD-driven, hands-free turns
  • M4 Brain — a real tool, a persona, sentence-streamed TTS
  • M5 Survivor — native speech-to-speech, barge-in, watchdog
  • Graded on behavior: if the TA can talk to it, you pass
The inversion: most courses make you build plumbing for a month before anything interesting happens. Here the agent speaks in week 1, and every later week answers "why does the naive version fall short?"

Where voice agents live today

  • Call centers — the biggest deployment surface
  • Vertical agents: property management, healthcare, restaurants, logistics
  • Assistants: in-car, smart speakers, phones
  • Most business calls are routine and structured — ideal targets
Use property management as the running example all course: leasing inquiries, and the tenant calling about a broken heater at 2am.

Why voice is hard (vs. a chatbot)

  • No undo — spoken words can't be edited after delivery
  • Latency is visceral — 3 s of phone silence feels broken
  • Ambiguity — no punctuation, homophones, noise, accents
  • Turn-taking — humans interrupt, backchannel, pause mid-thought
The chatbot mental model actively misleads. This whole course exists because of these four bullets.

The latency budget

  • Human turn gap: ~200–300 ms
  • "Feels responsive": under ~1 s mouth-to-ear
  • Budget = network + buffering + endpointing + inference + synthesis + playout
  • Every component in this course is fighting for milliseconds
Draw it as a stacked bar. Endpointing usually costs more than model inference — you must wait through silence to know they're done. That fight is week 3.

Architecture #1: the cascade

  • Mic → STT → LLM → TTS → speaker
  • Text is the interface between every stage
  • Modular, debuggable, swappable — transcripts for free
  • But: latency stacks per stage; tone and hesitation are lost
This is what students build in week 4. "Text as interface" is why cascades are debuggable AND why they're lossy.

Architecture #2: native speech-to-speech

  • One multimodal model — audio in, audio out, one WebSocket
  • xAI Grok Voice Agent API (ours) · Gemini Live · OpenAI Realtime
  • The OpenAI Realtime protocol is a de facto standard — xAI speaks it
  • Lower latency, hears tone; less control, provider holds state, quotas
Learn the event model once (week 5) and you can drive xAI, OpenAI, and — with small diffs — Gemini. These APIs still emit text transcripts of both sides: always keep them for debugging.

Cascade vs. native — when?

  • Compare on: latency · control · debuggability · cost · voice quality
  • Cascade → control and auditability
  • Native → when conversational feel is the product
  • A well-built agent core works behind either
Foreshadow week 6: the same brain should serve voice, SMS, and email.

How audio travels: the phone path

  • Caller → carrier → telephony provider → WebSocket to your server
  • Audio arrives as JSON: base64 8 kHz µ-law, ~20 ms chunks, 50/sec
  • Control events ride the same socket: start, stop, mark, clear
Students are shocked a "phone call" reaches Python as JSON at 50 msgs/sec. Demo: decode one payload live in a REPL — "this is a phone call."

How audio travels: the browser path

  • getUserMedia → Web Audio → WebSocket → Python
  • Same architecture, no carrier — which is why the project uses it
  • Browser mic ≈ caller · WebSocket ≈ media stream, 1:1
Everything they build transfers to real telephony almost unchanged. That's the course design.

A production topology

  • Provider → WS handler → preprocess (decode · resample · denoise)
  • → VAD / turn detection → model session → paced sender → provider
  • Sidecars: transcripts, database, schedulers, human handoff
  • This diagram is the course's table of contents
Walk left to right, tag each block with its week number. Most important slide of the lecture — tell students to photograph it.

Everything is a stream

  • No request/response — long-lived concurrent streams
  • One call ≈ many asyncio tasks: recv, send, process, monitor
  • Python's answer: one event loop, cooperative multitasking
  • Golden rule (week 5): never block the event loop
Quick poll: who has used asyncio? Reassure — scaffolding handles the boilerplate; concepts matter.

Your toolkit: the xAI API in 90 seconds

  • One key, one base URL: https://api.x.ai/v1 · key from console.x.ai
  • OpenAI-SDK compatible: OpenAI(base_url="https://api.x.ai/v1")
  • Your three calls this week: STT (audio → transcript) · Grok (transcript → reply) · TTS (reply → audio)
  • Right now: everyone creates a key and runs a 3-line Grok completion
Do key setup IN CLASS — key/billing problems are the #1 homework blocker; find them now, not Sunday night. Snippet on screen and in the scaffold README. Check the models page live for current grok-4-family names.

Milestone 1: Talkbox

  • A push-to-talk Grok agent: hold, ask, release — hear Grok answer
  • Scaffold given: record button + playback page, server with an answer(audio) stub
  • You write the cascade: clip → xAI STT → Grok (+ short system prompt) → xAI TTS → audio back
  • Path: make the stub echo first (~10 min plumbing test), then swap in the three calls
  • Instrument it: log per-stage wall-clock time — STT / Grok / TTS
Echo-first de-risks the plumbing before the API. The timing logs are the pedagogical core: their own numbers become the latency budget made personal — and week 4's streaming payoff is measured against this baseline.

  • Deliverable: your latency breakdown (STT / Grok / TTS in ms) + which stage surprised you
  • "Talkbox is push-to-talk — list everything that must become automatic before it's a phone call"
  • Skim: a provider's media-stream docs; the xAI Voice Agent event list

WEEK 02

Audio for Engineers Who Skipped DSP

MILESTONE M2 · PLUMBER — the full audio pipeline

Sound becomes numbers, numbers become bugs

After today, int16, 8000 Hz, and µ-law will never scare you again.

Everything is a numpy array. The least glamorous, most load-bearing week — every later "sounds weird" bug traces back here.

Sampling & quantization

  • Rates in the wild: 8 k phone · 16 k speech models · 24 k TTS · 44.1 k music
  • Nyquist in one line: you capture up to half your sample rate
  • int16 PCM is the lingua franca (−32768 … 32767)
  • One second of phone audio = 16 KB
Show a 50 ms waveform zoomed until samples are visible dots. An audio buffer is an int16 array; that's it.

Why phones sound like phones

  • The network standardized on 8 kHz / G.711 in the 1970s — still the floor
  • ~4 kHz ceiling: fine for speech, bad for "s" vs "f"
  • Models want 16 k+ → upsample everything off a phone line
  • Your model literally hears worse on the phone path
Play the same sentence at 44.1 k, then 8 k. The class HEARS the week's motivation.

µ-law: 1972 compression you still handle

  • 16-bit linear → 8-bit logarithmic (hearing is log — spend bits on quiet)
  • Halves bandwidth; still the default telephony stream encoding
  • First ops on inbound audio: base64-decode, then µ-law-decode
  • Wrong order = loud static — a rite of passage
Show the companding curve. In Python it's a 256-entry lookup table or a library one-liner.

Frames: audio arrives in 20 ms pieces

  • 160 samples @ 8 kHz, steady cadence — 50 messages/sec
  • VADs hard-require exact 10/20/30 ms frames
  • Buffer partial frames until you have a full one
  • Real-time audio is a metronome, not a firehose
Cadence over size — this sets up the fixed-rate sender in week 5.

Resampling: 8 k ↔ 16 k ↔ 24 k

  • The rate zoo in one call: phone 8 k → model 16 k → TTS 24 k → phone 8 k
  • Naive drop/duplicate aliases — sounds metallic
  • Use a streaming resampler (e.g. soxr) as a black box
  • Persist resampler state across chunks — or get periodic clicks
Say the chunk-boundary gotcha twice. It will be half of the M2 bugs.

Bytes, endianness, and your future bugs

  • int16 little-endian vs float32 [−1, 1] — browsers/ML float, telephony int
  • Convert deliberately; overflow wraps and screeches
  • Debug ritual: write the buffer to .wav and look at it (Audacity)
Demo: take a "broken screeching" buffer, open in Audacity, diagnose byte misalignment in 10 seconds. Saves more student-hours than anything else this week.

Anatomy of a media-stream message

  • {"event":"media","media":{"payload":"<base64 µ-law>"}}
  • Same socket: start · stop · mark · clear
  • Outbound is the same shape in reverse
  • Our browser project mirrors this format on purpose
Live: base64 → µ-law → numpy → matplotlib. Fifteen lines from "phone call" to waveform. The demo that makes the stack graspable.

Jitter and the playout buffer

  • Networks deliver unevenly; speakers need perfect evenness
  • Accumulate 40–100 ms before starting playback
  • The course's central tradeoff: latency vs. robustness
This tradeoff reappears in endpointing (wk 3) and pacing (wk 5). First appearance in concrete form.

Cleaning the signal: AEC · NS · AGC

  • AEC echo cancellation — subtract what you played (critical for barge-in)
  • NS noise suppression — neural ≫ classical on non-stationary noise
  • AGC gain control — normalize volume
  • Off-the-shelf (WebRTC APM); order matters: AEC → NS → VAD
Play before/after of neural NS over faucet/street noise. You want the CLEANED signal feeding the speech detector.

The full pipeline, assembled

  • In: base64 → µ-law decode → PCM 8 k → resample 16 k → clean → VAD → model
  • Out: model 24 k → 8 k → µ-law encode → base64 → JSON
  • Every arrow: a place to log, buffer, and get bytes wrong
  • This diagram is the M2 spec

The best pipeline is no pipeline

  • The xAI Voice Agent API accepts audio/pcmu — µ-law 8 kHz, directly
  • Phone ↔ Grok bridge: relay base64 both ways, zero transcoding
  • Why learn all this anyway? Browsers don't speak µ-law; the cascade needs PCM; production debugging is this material
  • Principle: negotiate one format end-to-end > converting in the middle
Real production pattern: a Telnyx↔xAI bridge where both sides speak µ-law is nearly a pure relay. Only possible because someone knew enough codecs to configure both ends.

Milestone 2: Plumber

  • Upgrade Talkbox: clip-upload → continuous streaming over a WebSocket (button now gates the stream)
  • Route mic audio through a simulated phone line before STT: → int16 → 8 kHz µ-law → back up
  • Buffered playout for the TTS response — 20 ms framing, no clicks
  • Done = still answers questions, and you report STT accuracy with vs without the phone line
The accuracy comparison makes the codec lesson concrete: students SEE the transcript degrade through the phone line — exactly what production phone agents live with. Diagnosis chart: screech = scaling/endianness · rhythmic clicks = chunk-boundary resampling.

Homework

  • "Outbound audio clicks every 20 ms — likely cause and fix?"
  • "Playout buffer at 0 ms? At 500 ms? What happens, and why?"
  • Deliverable: one sentence transcribed clean vs through your phone line — both transcripts
  • Optional: the G.711 Wikipedia page (short, genuinely good)
WEEK 03

Turn-Taking: VAD, Endpointing & Interruptions

MILESTONE M3 · LISTENER — speech detection & turn boundaries

The hardest problem nobody talks about

Your agent doesn't just need to understand speech. It needs to know when to shut up and when to start.

Open with two failure recordings: an agent that interrupts mid-sentence, and one that leaves 4 s of dead air. Both had perfect speech recognition. Turn-taking is a separate axis of quality.

What is a "turn"?

  • Humans exchange turns in ~200 ms — faster than reaction time: we predict turn ends
  • Backchannels ("uh-huh") are not turn claims
  • Mid-thought pauses ("my address is… four two…") are not turn ends
  • Silence cannot define a turn — yet silence is what we measure
Everything this week is a coping strategy for that gap.

VAD: three generations

  • Energy threshold — "is it loud?" — fails on any noise
  • Classical (webrtcvad) — fast, tiny, tunable aggressiveness
  • Neural (Silero) — dramatically better, still cheap on CPU
  • Production pattern: cheap VAD as fast gate, neural for real decisions
Live demo Silero's speech-probability meter on the room mic: speech vs claps vs keyboard. Don't spend neural inference on obvious silence.

VAD errors and their costs

  • False positive: agent halts for a door slam; garbage streams to the model
  • False negative: agent talks over the caller; first words clipped
  • Tuning = choosing which error you prefer
  • Phone agents lean patient: talking over a caller is worse than a pause

Endpointing: deciding the user finished

  • Track speech → silence; wait N ms of continuous silence; commit
  • Literal xAI turn_detection fields: threshold · silence_duration_ms · prefix_padding_ms · idle_timeout_ms
  • Gemini Live: same ideas, different names — learn concepts, map names
  • Every ms of required silence is added to every response
Model inference might take 400 ms — a conservative 800 ms endpoint delay doubles perceived latency before the model even starts. In week 5 students set these exact JSON fields.

The patience dial

  • Eager: snappy — but splits "I live at…" into two turns
  • Patient: never interrupts — but dead air after every turn
  • Tune per audience: elderly callers, non-native speakers, digit strings → patient
  • Advanced: adapt dynamically mid-number, mid-list
Deliberately product-thinking, not code. This knob changes user satisfaction more than model choice does.

Semantic endpointing (horizon)

  • Silence-based endpointing is blind to content
  • Use the transcript-so-far: "I live at" is obviously unfinished
  • Native models increasingly do this internally
  • Still imperfect: "Tuesday" (done) vs "Tuesday…" (more coming)

Barge-in: the user interrupts the agent

  • Detect → stop playback immediately → flush every buffer → cancel in-flight response → listen
  • Target: silent within ~100–300 ms or it feels like being talked over
  • Wire shape: model sends input_audio_buffer.speech_started → you flush and send clear downstream
  • That two-line reaction is barge-in
Walk the buffer chain: model → server queue → WebSocket → provider/browser buffer → speaker. Miss any one flush and the agent "keeps talking."

The echo problem: your agent hears itself

  • Speakerphone echo + naive barge-in → the agent interrupts itself, forever
  • Fix 1: AEC — subtract the known playback signal
  • Fix 2: state gating — agent_is_speaking raises the evidence bar
  • Production uses both
The self-interrupting agent is a hilarious, real failure mode — war story slot. Turn-taking is a small state machine; bugs are missing transitions.

Don't stream silence

  • Between turns the mic still produces frames — of silence
  • Streaming them wastes quota and money, and confuses endpointing
  • VAD-gated streaming + prefix padding so first syllables survive
  • A 10-minute call is mostly silence — silence is money

The turn-taking state machine

  • IDLE · USER_SPEAKING · AGENT_SPEAKING · INTERRUPTED
  • Transitions = events: VAD onset/offset, playback start/finish, barge-in
  • Every handler is one of these transitions
  • "If you can't place a bug on this diagram, you don't understand it yet"
Build it live on the whiteboard with class input, transition by transition. It becomes their M3/M5 design doc.

Milestone 3: Listener

  • Drop the button. VAD + your endpointing logic now decide when turns start and end
  • On end-of-turn: trimmed utterance (+ prefix padding) → the cascade, automatically
  • Log it live: [SPEECH END after 3.2 s → sending to Grok]
  • Done = a hands-free multi-turn conversation — mid-sentence pauses don't lose the turn
  • Noise must not trigger Grok calls — every false trigger costs you money
The week the project starts feeling like magic: the button was the last piece of manual turn-taking, and removing it is the whole lecture made real. The false-trigger-costs-money line lands — use it.

Homework

  • "Agent responds before users finish their phone number — which knobs, at what cost?"
  • "Describe the barge-in buffer-flush chain in your architecture"
  • Warn: exact VAD frame sizes; missing prefix padding clips syllables; test in a noisy room
WEEK 04

The Brain: Grok, Tools & Prompting for Speech

MILESTONE M4 · BRAIN — tools + persona + streaming

Hands, a name, and half your latency back

Your agent already talks. This week it gets smart.

Demo finished M4 up front: a hands-free question, a Grok answer, then trigger the tool ("what's the status of my maintenance request?") with the tool call visible in server logs.

The cascade you've been building

  • Three weeks in: mic → VAD/endpointing → xAI STT → Grok → xAI TTS → buffered playout
  • You built every box in that diagram
  • Text is the interface → the transcript is your debugger
  • The brain unit-tests without any audio at all
A recap that doubles as a confidence moment — put the week-1 topology back up and check off what they've built. Testability is why production keeps cascades around even with native speech-to-speech available.

Where your milliseconds went

  • The class's own M1 latency logs: STT + Grok + TTS, run sequentially
  • Worst offender: waiting for the whole Grok reply before TTS can start
  • Today's fixes: streaming (next slide) + keeping tools fast
  • Week 5 fixes the rest: one model, one socket, no text bottleneck
Use real numbers students submitted with M1 — nothing motivates like their own logs. Week-1 latency-budget theory becomes their measured reality; streaming is the payoff.

Streaming: never wait for the whole answer

  • Non-streaming: TTS can't start until the last token — worst case
  • Streaming: chunk into sentences → TTS each as it completes
  • Playback starts after the first sentence, not the last
  • Typically cuts perceived latency by more than half
Draw both timelines — the visual makes it obvious. Subtlety: you now own an ordered queue of TTS jobs. Concurrency arrives uninvited.

Function calling: giving the brain hands

  • Describe functions (name, JSON-schema params, description)
  • Model replies "call this, with these args" → you execute → return → it continues
  • The description IS the interface — the model decides from your words
  • Grok does parallel calls: return all results before continuing
Live A/B: a vaguely-described tool the model never calls vs a precise one it calls reliably. Tool descriptions deserve more editing than system prompts.

Tool design for voice

  • Return data, not prose: {"status":"success","appointment":…}
  • Errors are data, not exceptions — the model recovers in speech ("that day's full — Tuesday?")
  • Keep tools <1 s, or say something first — dead air feels like a dropped call
  • Idempotency: callers repeat themselves; book_appointment will fire twice

Prompting for the ear, not the eye

  • No markdown — it's read aloud: "asterisk asterisk important asterisk asterisk"
  • Short sentences; front-load the answer; 1–3 sentences per turn
  • Numbers and addresses written as spoken; confirm digit strings back
  • Persona: name, role, limits, when to hand off to a human
Read a markdown response aloud verbatim. Gets a laugh; makes the point permanently.

State and memory

  • Three layers: the turn (prompt) · the conversation (history you resend) · the world (database, via tools)
  • Rule: anything that must survive a dropped call → database
  • Long calls: trim or summarize old turns
  • Load history at call start: "welcome back — about yesterday's leak?"
A call can die any second. What must the agent remember when they call back? That list = your database schema.

Multi-agent architectures

  • One do-everything prompt does nothing well
  • Thin triage/router → domain specialist with its own prompt, tools, history
  • Voice handoff: transfer or swap session config; text: persist the route
  • Channel adapters: same brain over voice, SMS, email
Running example: "I want to see the 2-bed on Main St" vs "my heater is broken" — different tools, urgency, prompts.

Guardrails for a live call

  • No human review between generation and delivery
  • Prompt-level: forbidden topics; tools as source of truth; disclose being AI
  • System-level: duration caps, loop caps, filters, human escape hatch
  • The prompt is a request; the system enforces
Don't tell the model to never quote a price — don't GIVE it prices except through a tool that returns approved ones.

Milestone 4: Brain

  • One real tool via Grok function calling (e.g. check_apartment_availability) + history across turns
  • A named persona with spoken-style constraints — no markdown reaches TTS
  • Sentence-streamed TTS: first audio out before generation finishes
  • Done = hands-free 3+ turns, one visible tool call, latency measurably below your M1 baseline
  • Stretch: second tool with parallel calls; "hold the floor" filler for slow tools
Streaming is core, not stretch — students have carried their M1 numbers for three weeks; this is where they beat them. Voluntary latency leaderboard: end-of-speech → first audio.

Homework

  • "Your tool takes 4 s — two ways to keep the call alive, with tradeoffs"
  • "Why must book_appointment be idempotent in voice, specifically?"
  • Required skim: the Voice Agent API event list — next week uses all of it
WEEK 05

Going Native: The Voice Agent API & Reliability

MILESTONE M5 · SURVIVOR — speech-to-speech + chaos-proofing

Going native (and staying alive)

  • Part 1: replace three API calls with one WebSocket
  • Part 2: everything fails mid-sentence — build for it
Four weeks of cascade isn't wasted: it's what you debug with, fall back to, and use on text channels. And weeks 1–4 are about to reappear as literal JSON fields.

The Voice Agent API mental model

  • wss://api.x.ai/v1/realtime?model=grok-voice-latest
  • Typed JSON events, both directions
  • Up: input_audio_buffer.append · down: response.output_audio.delta
  • STT, VAD, endpointing, LLM, TTS all server-side — you configure instead of build
Show a raw event-type log of a 20-second session scrolling by. Students should recognize the anatomy already.

The handshake — order matters

  • Connect → conversation.created
  • You send session.update → server confirms session.updated
  • Only now stream audio — gate your send loop on a ready flag
  • Greet via one-off response instructions, not a fake user message
Audio sent early is processed with wrong settings. And greeting via fake user message makes the model echo and elaborate; one-off instructions make it say the line once. Small protocol choices, audible differences.

session.update: weeks 2–3 as JSON

  • instructions · voice (eve, ara, rex, sal, leo, custom)
  • audio.*.format: audio/pcm at your rate — or audio/pcmu, the phone codec
  • turn_detection: server VAD with the week-3 knobs, verbatim
  • Extras: transcription language_hint + keyterms · pronunciation replace · output speed · reasoning.effort
The payoff slide: nothing new here — the course rendered as config. Match formats end-to-end to skip pointless resampling. keyterms fixes domain vocabulary (unit codes, street names).

Barge-in, for real this time

  • Server VAD emits input_audio_buffer.speech_started
  • Your job: stop playback, flush every downstream buffer, drop queued deltas
  • The model stops generating — you stop playing
  • Test: interrupt mid-monologue → silent in ~100–300 ms
The event does detection; the reaction is still all your code, and a missed buffer is audible. M5 grading item — the TA will interrupt.

Function calling over the wire

  • response.function_call_arguments.done → name, call_id, args
  • Execute → conversation.item.create (function_call_output) → response.create
  • The double-reply bug: one turn, many calls — trigger a response per tool and the model speaks once per tool
  • Fix: submit all outputs; exactly one response.create on response.done
Tell it as a war story: "the agent answered everything twice until we counted response.creates." Deeper lesson: in event protocols, correctness lives in WHEN you send.

Scripted speech: force_message

  • Some lines must be exact: legal disclosures, verification prompts
  • Straight TTS, zero model involvement; optionally interruptible: false
  • Pattern: the model speaks the empathy; force_message speaks the compliance
  • A force_message is the turn — don't follow it with response.create
LLMs paraphrase; some words legally can't be. Follow it with response.create and you get bonus improvisation.

Part 2: everything fails mid-sentence

  • Taxonomy: model stalls · WS drops · client side drops · your loop blocks
  • The caller hears silence for all of them
  • Detect fast → react in order of severity → never leave dead air
  • "A voice agent's uptime is measured per-sentence"

Watchdogs

  • Timestamp every model event; background task checks recency
  • Context-aware: "user finished, no response yet" ≠ "no events at all"
  • False-positive trap: silence during your own playback is normal — the watchdog needs the turn-taking state machine
  • Escalate: verbal nudge → reconnect → graceful goodbye — never just go silent
A naive watchdog fires during long agent utterances and "recovers" a healthy call into a broken one. Keep thresholds as "tune for your product" — the structure is the lesson.

Reconnection without amnesia

  • Dropped model WS ≠ dropped call — reconnect with backoff, buffer caller audio meanwhile
  • Fresh session = no memory. xAI's answer: session resumption
  • Enable it, capture conversation.id, reconnect with ?conversation_id=…
  • The cascade just resends its own history — a real architectural tradeoff
Close the loop to week 4's state layers: a booked appointment must survive even a FAILED resumption — that's why the world lives in your database.

Don't block the loop

  • One 200 ms synchronous call stutters every concurrent call
  • Await everything; CPU work in C-extensions or executors
  • Pace outbound audio from a buffer — smooth playback, instant flush
  • Log every event with timestamps — you can't debug what you didn't capture
If time allows: add time.sleep(0.2) to a handler and let them HEAR the glitch.

Milestone 5: Survivor

  • Swap the cascade for a Voice Agent session; keep the week-4 tool working
  • Barge-in works — the TA will interrupt
  • Watchdog survives a simulated mid-call WS kill (chaos.py provided)
  • Done = conversation + interruption + audible failure recovery
Watching your own agent survive a chaos test is the most satisfying moment of the course.

Homework

  • "Watchdog fires mid-sentence on a healthy call — what state was it missing?"
  • "Cascade vs native: compare disconnect recovery. Which do you trust?"
  • Optional: ephemeral client secrets — browser-direct sessions, no server in the audio path

WEEK 06

Production, the Provider Landscape & Demo Day

≈45 MIN LECTURE + ≈45 MIN DEMOS

From demo to deployment

Your agent works on your laptop. Today: what stands between that and a phone number real people call.

Keep the lecture tight — the class wants to demo. Every slide is a pointer into a rabbit hole, not the rabbit hole.

Putting it on a real phone line

  • Rent a number → webhook on call → answer + start a bidirectional media stream
  • Base64 µ-law over WS — exactly as week 2 promised
  • Reference bridge: phone → Telnyx → your server ↔ xAI Voice Agent WS
  • With audio/pcmu: nearly a pure relay; barge-in = speech_started → clear
The whole bridge is a few hundred lines of Python. Everything browser-based they built maps 1:1 — that was the course design, not luck. Call control adds transfer, hangup, SMS follow-up.

One brain, many channels

  • Real products answer the phone AND the texts AND the email
  • Domain logic once; thin channel adapters per transport
  • Voice = realtime speech-to-speech; text channels = chat completions, same tools
  • State keyed by person, not channel
Example: tenant calls about a leak → plumber scheduled → confirmation by SMS → survey by email. One conversation, three transports.

The landscape: Gemini Live & OpenAI Realtime

  • You know xAI's version — the others are a diff, not a new subject
  • Gemini Live: Google SDKs; VAD = start/end sensitivity + prefix padding + silence duration (week-3 names, Google spelling); built-in context compression; regional endpoints
  • OpenAI Realtime: the protocol your M5 already speaks — ~90% portable
  • Real differentiators: voice quality · tools · resumption · price · concurrent-session quota — the production ceiling
Teach comparatively: map every Gemini concept to the xAI feature they used. Side-by-side config snippets if time allows. Teams plan capacity around CPUs and discover the ceiling is session quota — ask early.

Evaluating a voice agent

  • "It sounded fine to me" doesn't scale
  • Metrics: task completion · handoff rate · latency percentiles · barge-in frequency
  • Transcript sampling + LLM-as-judge rubrics
  • Prompts are code: scripted-persona regression evals on every change
  • Audio layer too: domain-vocabulary WER, VAD-in-noise
Serious teams run synthetic personas (slow talker, topic-changer, mumbler) against every change before shipping. Great final-writeup topic.

Cost of a voice minute

  • Stack the meters: telephony + model session time (or STT+LLM+TTS) + infra
  • Cascade vs native is also a cost choice — changes with volume and call length
  • Silence is money: week 3's VAD gating cuts the bill directly
  • In-class exercise: estimate a 5-minute call, both architectures, live pricing pages
The method (enumerate meters × minutes) is the lesson — printed prices go stale within a quarter.

Compliance & safety

  • Recording consent varies by jurisdiction — disclose, always
  • AI disclosure via scripted delivery (force_message)
  • PII fills transcripts: encrypt, minimize retention, redact logs; quiet hours outbound
  • "I smell gas" → deterministic escalation that bypasses the LLM entirely
Emergency routing must be system code, not model judgment — week 4's "prompt is a request, system enforces" at its highest stakes.

Scaling — one slide of systems reality

  • One non-blocking asyncio process goes surprisingly far
  • Externalize state before adding workers; mind WS stickiness
  • Instance-level horizontal scaling sidesteps multi-worker pain
  • The ceiling is usually provider quota, not your CPUs

Where the field is going

  • Full-duplex models — listening while speaking, real backchannels
  • Semantic endpointing by default; VAD as a hint
  • Expressive TTS + custom voice cloning (already shipping)
  • Speech-native reasoning; on-device voice
End where week 1 began: the 200 ms human turn gap. The industry is clawing toward it — your students now know exactly which parts are hard, and why.

Demo day

  • 5 min: 30-sec pitch → live conversation with one tool call + one barge-in → your worst failure and its fix
  • Rubric: works live 40% · turn-taking 20% · tools 20% · failure story 20%
  • Audience: one "I'd steal that" note per demo
The failure segment is deliberately weighted — it produces the best learning and the best stories. Backup plan for Wi-Fi: hotspot + recorded fallback with live Q&A.
