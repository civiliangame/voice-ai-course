WK 1The Anatomy of a Voice Agent

Why latency dominates everything; the two dominant architectures (cascade vs. native speech-to-speech); how audio physically travels from a caller to a model and back — plus an in-class xAI quickstart, because the week-1 homework is already a talking Grok agent.

M1 Talkbox75–90 minin class: xAI key @ console.x.ai
Full slide deck (15 slides)

01Voice Is the Hardest UI

  • Hook: “By week 6 you'll have built an AI you can talk to — and interrupt.”

Note: open with a 30-second live demo of the finished project, including an interruption. Then: “everything you just heard is a lie held together by buffering.”

02What You'll Build

  • The milestone ladder, one sentence each
  • Graded on behavior: if the TA can talk to it, you pass

03Where Voice Agents Live Today

  • Call centers; vertical agents (property management, healthcare, restaurants); assistants
  • Most business calls are routine and structured — ideal automation targets

Note: pick one vertical (property management works well: leasing calls, 2am maintenance emergencies) as the running example all six weeks.

04Why Voice Is Hard (vs. a Chatbot)

  • No undo — spoken words can't be edited after delivery
  • Latency is felt viscerally: 3 s of phone silence feels broken
  • Ambiguity: no punctuation, homophones, noise
  • Turn-taking: humans interrupt, backchannel, pause mid-thought

05The Latency Budget

  • Human turn gap: ~200–300 ms; “feels responsive” target: under ~1 s mouth-to-ear
  • Stacked-bar budget: network + buffering + endpointing + inference + synthesis + playout

Note: endpointing usually costs more than model inference — you must wait through silence to know the user finished. That fight is week 3.

06Architecture #1: The Cascade

  • Mic → STT → LLM → TTS → speaker; text is the interface between stages
  • Modular, debuggable, swappable — but latency stacks and tone is lost

07Architecture #2: Native Speech-to-Speech

  • One multimodal model, audio in / audio out, over a bidirectional WebSocket
  • Examples: xAI Grok Voice Agent API (our platform), Gemini Live, OpenAI Realtime
  • The OpenAI Realtime wire protocol is a de facto standard — xAI is compatible with it
  • Lower latency, hears tone; less controllable, provider holds session state, quota limits

Note: learn the event model once (week 5) and you can drive xAI, OpenAI, and — with small diffs — Gemini.

08Cascade vs. Native: When to Use Which

  • Compare: latency / control / debuggability / cost / voice quality
  • Rule of thumb: cascade for control and auditability; native when conversational feel is the product
  • A well-built agent core works behind either

09How Audio Travels: The Phone Path

  • Caller → carrier → telephony provider → WebSocket to your server
  • Audio arrives as JSON: base64 8 kHz µ-law, ~20 ms chunks, 50 msgs/sec
  • Control events on the same socket: start, stop, mark, clear

Note: show one redacted media message; decode its payload live in a REPL — “this is a phone call.”

10How Audio Travels: The Browser Path (Our Project)

  • getUserMedia → Web Audio → WebSocket → Python — same shape, no carrier
  • Draw the 1:1 mapping onto the phone path

11A Production Topology

  • Provider → WS handler → preprocessing (decode/resample/denoise) → VAD → model session → paced sender → provider
  • Sidecars: transcripts, database, schedulers, human handoff

Note: tag each block with its week number — this diagram is the course's table of contents. Students should photograph it.

12Everything Is a Stream

  • No request/response: long-lived concurrent streams; one call = many asyncio tasks
  • Golden rule preview: never block the event loop (week 5)

13Your Toolkit: The xAI API in 90 Seconds

  • https://api.x.ai/v1, Bearer auth, key from console.x.ai; OpenAI-SDK compatible
  • The three calls for this week's homework: STT · Grok chat · TTS
  • Live, in class: everyone creates a key and runs a 3-line Grok completion

Note: key/billing problems are the #1 homework blocker — find them now, not Sunday night.

14Milestone 1: Talkbox

  • Push-to-talk Grok agent: hold, ask, release — hear Grok answer out loud
  • Scaffold given (record button + playback + server stub); you write the cascade: clip → xAI STT → Grok → xAI TTS

  • Instrument it: log per-stage wall-clock time (STT / Grok / TTS)

WK 2Audio for Engineers Who Skipped DSP

PCM, sample rates, µ-law, framing, resampling, playout buffers. The least glamorous, most load-bearing week — every later “sounds weird” bug traces back here.

M2 Plumber75–90 min
Full slide deck (14 slides)

01Sound Becomes Numbers, Numbers Become Bugs

Note: promise: after today, int16, 8000 Hz, and µ-law never scare them again. It's all numpy arrays.

02Sampling & Quantization

  • Rates in the wild: 8 k phone · 16 k speech models · 24 k TTS · 44.1 k music
  • Nyquist in one line; int16 PCM as the lingua franca
  • 1 second of phone audio = 16 KB

03Why Phones Sound Like Phones: 8 kHz & G.711

  • The 1970s standard is still the floor; ~4 kHz ceiling — bad for “s” vs “f”
  • Models want 16 k+ → upsample everything off a phone line

Note: play the same sentence at 44.1 k then 8 k — the class hears the week's motivation.

04µ-law: 1972 Compression You Still Must Handle

  • 16-bit linear → 8-bit logarithmic (hearing is log — spend precision on quiet)
  • First op on inbound phone audio: base64-decode, then µ-law-decode. Wrong order = loud static

05Frames: Audio Arrives in 20 ms Pieces

  • 160 samples @ 8 kHz, steady 50/sec cadence; VADs require exact 10/20/30 ms frames
  • Real-time audio is a metronome, not a firehose

06Resampling: 8 k ↔ 16 k ↔ 24 k

  • The rate zoo in one call; naive drop/duplicate aliases and sounds metallic
  • Use a real streaming resampler (e.g. soxr); persist state across chunks or get periodic clicks

Note: say the chunk-boundary gotcha twice — it will be half of the M2 bugs.

07Bytes, Endianness, and the Bugs You'll Meet

  • int16-LE vs float32 [−1,1]: browsers/ML use float, telephony uses int — convert deliberately
  • Debug ritual: write the buffer to .wav and look at it in Audacity

08Anatomy of a Media-Stream Message

  • {"event":"media","media":{"payload":"<base64 µ-law>"}} plus start/stop/mark/clear
  • Our browser project mirrors this format on purpose

09Jitter and the Playout Buffer

  • Networks deliver unevenly; speakers need perfect evenness → buffer 40–100 ms
  • The course's central tradeoff appears: latency vs. robustness

10Cleaning the Signal: AEC · NS · AGC

  • Echo cancellation (critical for barge-in), noise suppression (neural ≫ classical on non-stationary noise), gain control
  • Off-the-shelf pipeline stage (WebRTC APM); order matters: AEC → NS → VAD

11The Full Pipeline, Assembled

  • In: base64 → µ-law decode → PCM 8 k → resample 16 k → clean → VAD → model
  • Out: model 24 k → 8 k → µ-law → base64 → JSON
  • Every arrow is a place to log, buffer, and get bytes wrong — this diagram is the M2 spec

12The Best Pipeline Is No Pipeline

  • The xAI Voice Agent API accepts audio/pcmu (µ-law 8 kHz) directly
  • A phone↔Grok bridge can relay base64 µ-law both ways: zero transcoding
  • We build the pipeline anyway: browsers don't speak µ-law; the cascade needs PCM; production debugging is this material

Note: design principle — negotiate a common format end-to-end instead of converting in the middle. Only possible if you know enough codecs to ask.

13Milestone 2: Plumber

  • Upgrade Talkbox from clip-upload to continuous WebSocket streaming (the button now gates the stream)
  • Route mic audio through a simulated phone line (8 kHz µ-law round-trip) before STT; buffered playout for the reply
  • Done: still answers questions, click-free — and you report STT accuracy with vs without the phone line

Note: students see the transcript degrade through the phone line — the codec lesson made concrete. Diagnosis: screeching = endianness/scaling; rhythmic clicks = chunk-boundary resampling.

14Homework

  • “Outbound audio clicks every 20 ms — most likely cause and fix?”
  • “What happens with a 0 ms playout buffer? A 500 ms one?”

44.1 k vs 8 k listening test · REPL decode of a media message to a waveform plot · Audacity debugging of a corrupted buffer · neural noise suppression before/after

WK 3Turn-Taking: VAD, Endpointing & Interruptions

Knowing when the user is talking — and when they're done — is its own hard subproblem, and the one that most determines whether an agent feels natural.

M3 Listener75–90 min
Full slide deck (13 slides)

01The Hardest Problem Nobody Talks About

Note: open with two failure recordings — an agent that interrupts, and one that leaves 4 s of dead air. Both had perfect speech recognition.

02What Is a “Turn”?

  • Humans exchange turns in ~200 ms — faster than reaction time; we predict turn ends
  • Backchannels (“uh-huh”) aren't turn claims; mid-thought pauses aren't turn ends

Note: punchline — silence can't define a turn boundary, yet silence is mostly what we measure. The week is coping strategies for that gap.

03VAD: Three Generations

  • Energy threshold → classical statistical (webrtcvad) → neural (Silero)
  • Production pattern: cheap VAD as fast gate, neural for decisions that matter

04VAD Errors and Their Costs

  • False positive: agent halts for a door slam. False negative: agent talks over the caller, first words clipped
  • Tuning = choosing which error you prefer; phone agents lean patient

05Endpointing: Deciding the User Finished

  • Wait N ms of continuous silence after speech, then commit the turn
  • These are literal xAI turn_detection fields: threshold, silence_duration_ms, prefix_padding_ms, idle_timeout_ms
  • Gemini Live: same ideas, different names — learn concepts, map names
  • Every ms of required silence is added to every response

Note: students will set these exact JSON fields in week 5 — flag that this slide is directly actionable.

06The Patience Dial

  • Eager: snappy but splits “I live at…” into two turns. Patient: never interrupts, dead air everywhere
  • Tune per audience; adapting dynamically (mid-number, mid-list) is the advanced move

07Semantic Endpointing (Horizon)

  • Use the transcript-so-far to judge completeness — “I live at” is obviously unfinished
  • Native models increasingly do this internally

08Barge-In: The User Interrupts the Agent

  • Detect → stop playback immediately → flush every buffer to the speaker → cancel in-flight response → listen
  • Go silent within ~100–300 ms or it feels like being talked over
  • Realtime-API shape: model sends input_audio_buffer.speech_started; you flush and send clear downstream — that reaction is barge-in

09The Echo Problem: Your Agent Hears Itself

  • Speakerphone echo → naive barge-in triggers on the agent's own voice → it interrupts itself, forever
  • Fixes: AEC + state gating (agent_is_speaking flag raises the evidence bar)

10Don't Stream Silence

  • VAD-gated streaming with prefix padding; silence costs quota, money, and endpointing accuracy

11The Turn-Taking State Machine

  • IDLE · USER_SPEAKING · AGENT_SPEAKING · INTERRUPTED, with events as transitions
  • “If you can't place a bug on this diagram, you don't understand it yet”

Note: build it live on the board with class input — it becomes their M3/M5 design doc.

12Milestone 3: Listener

  • Drop the button: VAD + endpointing decide turns; end-of-turn sends the trimmed utterance through the cascade automatically
  • Done: a hands-free multi-turn conversation — mid-sentence pauses don't split turns, noise doesn't trigger Grok calls

Note: the week the project starts feeling like magic — removing the button is the whole lecture made real. False triggers cost money; mention it, it lands.

13Homework

  • “Agent responds before users finish their phone number — which knobs, at what cost?”
  • “Describe the buffer-flush chain for barge-in in your architecture”
Live demos
Failure recordings · Silero speech-probability meter on the room mic · whiteboard state machine
Pitfalls to warn
webrtcvad hard-requires exact frame sizes · missing prefix padding clips first syllables · test in a noisy room, not just a quiet one

WK 4The Brain: Grok, Tools & Prompting for Speech

The agent already talks — this week it gets smart: function calling against real data, a persona, sentence-streamed TTS that halves latency, and prompting for the ear instead of the eye.

M4 Brain75–90 minbeat your M1 latency
Full slide deck (12 slides)

01Hands, a Name, and Half Your Latency Back

Note: demo finished M4 up front — hands-free question, Grok answer, visible tool call in the server logs.

02The Cascade You've Been Building

  • Three weeks in: mic → VAD → xAI STT → Grok → xAI TTS → buffered playout — students built every box
  • Text is the interface → the transcript is your debugger; the brain unit-tests without audio

Note: recap-as-confidence-moment: put the week-1 topology back up and check off what they've built.

03Where Your Milliseconds Went

  • The class's own M1 latency logs: STT + Grok + TTS run sequentially
  • Worst offender: waiting for the whole Grok reply before TTS starts — streaming fixes it today
  • Week 5 fixes the rest: one model, one socket, no text bottleneck

Note: use real numbers students submitted with M1 — nothing motivates like their own logs.

04Streaming: Never Wait for the Whole Answer

  • Sentence-chunked TTS: playback starts after the first sentence, not the last
  • Typically cuts perceived latency by more than half; now you own an ordered TTS queue

05Function Calling: Giving the Brain Hands

  • Describe functions (name, JSON-schema params, description) → model requests calls → you execute and return results
  • The description is the interface; Grok supports parallel calls — return all results before continuing

Note: live A/B a vaguely-described tool (never called) vs a precise one (called reliably).

06Tool Design for Voice

  • Return data, not prose; errors are data too — {"status":"error",…} lets the model recover in speech
  • Keep tools <1 s or hold the floor first; make book_appointment idempotent — callers repeat themselves

07Prompting for the Ear, Not the Eye

  • No markdown — it gets read aloud, asterisks and all; 1–3 sentence turns; front-load answers
  • Numbers and addresses written the way they're spoken; confirm digit strings back
  • Persona in the system prompt: name, role, limits, when to hand off

Note: read a markdown response aloud verbatim. Gets a laugh; makes the point permanent.

08State and Memory

  • Three layers: the turn (prompt) · the conversation (history you resend) · the world (database, via tools)
  • Rule: anything that must survive a dropped call goes in the database

09Multi-Agent Architectures

  • Triage/router classifies intent → domain specialist with its own prompt, tools, history
  • Channel adapters: same brain over voice, SMS, email — transport out of business logic

10Guardrails for a Live Call

  • No human review between generation and delivery
  • “The prompt is a request; the system enforces”: caps, filters, deterministic escalation, tools as source of truth

11Milestone 4: Brain

  • One real tool via function calling (e.g. check_apartment_availability) + history across turns + a named persona
  • Sentence-streamed TTS — first audio out before generation finishes
  • Done: hands-free 3+ turns, one visible tool call, latency measurably below the M1 baseline

Note: streaming is core this week, not stretch — students beat the M1 numbers they've carried for three weeks. Voluntary latency leaderboard.

12Homework

  • “Your tool takes 4 s — two techniques to keep the call alive, and their tradeoffs”
  • “Why must book_appointment be idempotent in voice specifically?”
  • Required skim: the Voice Agent API event list — next week uses all of it
Pitfalls to warn
Wrong base URL / unset key · TTS-ing raw markdown · synchronous API calls blocking the loop · sending untrimmed audio to STT

WK 5Going Native: The Voice Agent API & Reliability

Replace the cascade with a native speech-to-speech session over wss://api.x.ai/v1/realtime, master the event protocol, then add the layer that separates demos from products: watchdogs, reconnection, resumption.

M5 Survivor90 min — densest week
Full slide deck (13 slides)

01Going Native (and Staying Alive)

  • Part 1: replace three API calls with one WebSocket. Part 2: everything fails mid-sentence

Note: the cascade isn't wasted — it's what you debug with, fall back to, and use on text channels.

02The Voice Agent API Mental Model

  • wss://api.x.ai/v1/realtime?model=grok-voice-latest; typed JSON events both directions
  • Audio up via input_audio_buffer.append; speech down via response.output_audio.delta
  • STT, VAD, endpointing, LLM, TTS all live server-side — you configure instead of build

03The Handshake (Order Matters)

  • Connect → conversation.created → you send session.update → session.updated → only now stream audio
  • Gate the send loop on a ready flag — audio sent early is processed with wrong settings
  • Greet via one-off response instructions, not a fake user message (which the model echoes and elaborates)

04session.update: Weeks 2–3 as JSON

  • instructions, voice (eve · ara · rex · sal · leo · custom)
  • audio.*.format: audio/pcm at your rate, or audio/pcmu — the phone codec (week 2!)
  • turn_detection: server VAD with the week-3 knobs, verbatim
  • Extras: transcription language_hint + keyterms, pronunciation replace, output speed, reasoning.effort

Note: the payoff slide — nothing here is new, it's the whole course rendered as config. Match formats end-to-end to avoid pointless resampling.

05Barge-In, For Real This Time

  • On input_audio_buffer.speech_started: stop playback, flush every downstream buffer, drop queued deltas
  • The model stops generating; you stop playing. A missed buffer is audible

06Function Calling Over the Wire

  • response.function_call_arguments.done → execute → conversation.item.create (type function_call_output, same call_id) → one response.create
  • The double-reply bug: one turn can contain several calls; trigger response.create per tool and the model speaks once per tool. Submit all outputs, then exactly one response.create on response.done

Note: tell it as a war story. Deeper lesson: in event protocols, correctness lives in when you send, not just what.

07Scripted Speech: force_message

  • Exact-wording lines (legal disclosures, verification prompts): straight TTS, no model involvement, optionally uninterruptible
  • Pattern: the model speaks the empathy; force_message speaks the compliance
  • A force_message is the turn — don't follow it with response.create

08Part 2: Everything Fails Mid-Sentence

  • Taxonomy: model stalls · WS drops · client side drops · your own loop blocks
  • The caller hears silence for all of them — “uptime is measured per-sentence”

09Watchdogs

  • Track last-event timestamps; a background task checks recency against context-aware thresholds
  • “User finished N s ago, no response yet” ≠ “no events for M s” — different conditions, different reactions
  • False-positive trap: silence during your own playback is normal — the watchdog needs the turn-taking state machine
  • Escalate: verbal nudge → reconnect → graceful goodbye. Never just go silent

10Reconnection Without Amnesia

  • Dropped model WS ≠ dropped call: reconnect with backoff while buffering caller audio
  • xAI session resumption: enable it, capture conversation.id, reconnect with ?conversation_id=… and turns replay
  • Cascade owns its history and just resends — a real architectural tradeoff vs native

11Don't Block the Loop

  • One 200 ms synchronous call stutters every concurrent call on the loop
  • Await everything; CPU work in C-extensions/executors; pace outbound audio from a buffer (smooth playback + instant flush)
  • Log every event with timestamps — you can't debug what you didn't capture

12Milestone 5: Survivor

  • Swap the cascade for a Voice Agent session; keep the week-4 tool working over the wire protocol
  • Barge-in works (TA will interrupt); watchdog survives a simulated mid-call WS kill — recover or say goodbye, never dead air

Note: provide a chaos.py that randomly closes their model WebSocket. Watching your own agent survive a chaos test is the best moment of the course.

13Homework

  • “Watchdog fires mid-sentence on a healthy call — what state was it missing?”
  • “Compare cascade vs native recovery of conversation memory after a disconnect”
  • Optional: ephemeral client secrets — browser-direct sessions that cut your server out of the audio path

WK 6Production, the Provider Landscape & Demo Day

Real phone lines, the multi-provider landscape (Gemini Live enters here), evaluation, cost, compliance — then 45 minutes of student demos.

Full slide deck (10 slides + demos)

01From Demo to Deployment

  • “Your agent works on your laptop. Today: what stands between that and a phone number real people call.”

02Putting It on a Real Phone Line

  • Rent a number → webhook on incoming call → answer + start a bidirectional media stream (base64 µ-law over WS — exactly as promised in week 2)
  • Reference bridge: phone → Telnyx → your server ↔ xAI Voice Agent WS; with audio/pcmu it's nearly a pure relay
  • Barge-in on the phone: speech_started → provider clear; call control adds transfer, hangup, SMS follow-up

Note: the whole bridge is a few hundred lines of Python — and everything browser-based they built maps 1:1 onto it. That was the course design, not luck.

03One Brain, Many Channels

  • Voice + SMS + email, one conversation across transports; state keyed by person, not channel
  • Domain logic in one place; thin channel adapters (week 4's rule, at product scale)

04The Landscape: Gemini Live & OpenAI Realtime

  • Now that you know xAI's version, the others are a diff, not a new subject
  • Gemini Live: bidirectional streaming via Google's SDKs; VAD as start/end-of-speech sensitivity + prefix padding + silence duration (week-3 names, Google spelling); built-in context compression; regional endpoints matter
  • OpenAI Realtime: the protocol your M5 already speaks — ~90% portable by changing the URL
  • What actually differentiates: voice quality, tool ecosystems, resumption, price — and concurrent-session quota, the real production ceiling

Note: teach comparatively — map every Gemini concept back to the xAI feature students used. Side-by-side config snippets if time allows.

05Evaluating a Voice Agent

  • Task completion, handoff rate, latency percentiles, barge-in frequency, duration
  • Transcript sampling + LLM-as-judge rubrics; regression evals with scripted personas on every prompt change — prompts are code, with tests
  • The audio layer needs evals too: domain-vocabulary WER, VAD-in-noise

06Cost of a Voice Minute

  • Stack the meters: telephony + model session time (or STT+LLM+TTS) + infra
  • Silence is money — week 3's VAD gating cuts the bill
  • In-class estimate for a 5-minute call, both architectures, from live pricing pages

07Compliance & Safety

  • Recording consent varies by jurisdiction — disclose, always; AI disclosure via scripted delivery
  • PII everywhere in transcripts: encrypt, minimize retention, redact logs; quiet hours for outbound
  • Emergencies (“I smell gas”) get deterministic escalation that bypasses the LLM entirely

08Scaling (One Slide of Systems Reality)

  • One non-blocking asyncio process goes surprisingly far; externalize state before adding workers
  • Instance-level horizontal scaling sidesteps multi-worker pain; the ceiling is usually provider quota, not CPU

09Where the Field Is Going

  • Full-duplex models (listening while speaking) · semantic endpointing by default · expressive/custom voices · speech-native reasoning · on-device

Note: end where week 1 began — the 200 ms human turn gap. The industry is clawing toward it; your students now know which parts are hard and why.
